# 非機能要件 / 可用性 作成の指示テンプレート

以下のルールに従って、**非機能要件: 可用性（`nfr-availability`）** を 1 ファイル作成してください。出力は **Markdown** とします。

- 可用性は稼働率、許容停止時間、RTO/RPO を基準化します。
- 対象時間帯と除外条件を必ず明示してください。

## 事前に与えられる情報（入力）

- 対象システム、対象時間帯、計画停止条件
- 復旧目標（RTO/RPO）と業務影響度
- 検証先（`sac-*` / `sts-*`）と運用手順（`opd-*` / `opr-*`）

## 出力フォーマット（必須）

### YAML Frontmatter（必須）

```yaml
---
id: nfr-availability
type: architecture
title: 非機能要件: 可用性
status: draft # draft / ready / deprecated
part_of: [nfr-index]
based_on: []
supersedes: []
---
```

### 本文の見出し（必須・順序固定）

1. 概要（対象・目的）
2. 適用範囲・除外条件
3. 可用性要件一覧
4. 測定・検証方法
5. 関連ドキュメント導線

## 記述ルール（重要）

- 要件一覧は表形式（`id` / `内容` / `指標` / `基準` / `備考`）で記載する。
- 指標は月間稼働率、RTO、RPO などを使い、算出対象時間を明示する。
- 測定・検証方法は監視値、障害訓練、復旧試験で判定できる形にする。

## 禁止事項

- 稼働率の対象時間を示さない。
- RTO/RPO を定義しない。
- 「可能な限り復旧」など曖昧表現を使う。

## 最終チェック（自己検査）

- `id: nfr-availability` / `type` / `status` が設定されている
- 見出しが指定順序で揃っている
- 要件一覧に `指標` と `基準` がある
- 適用範囲に対象時間帯と除外条件がある
- 測定・検証方法に判定タイミングと判定者がある
