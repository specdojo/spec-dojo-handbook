# 非機能要件 / 性能 作成の指示テンプレート

以下のルールに従って、**非機能要件: 性能（`nfr-performance`）** を 1 ファイル作成してください。出力は **Markdown** とします。

- 性能は応答性・処理能力・資源効率を基準化します。
- 平常時・ピーク時の条件を分離し、再現可能に記述してください。
- 平均値より P95/P99 などの分位値を優先して評価します。

## 事前に与えられる情報（入力）

- 対象処理、業務価値、対象時間帯
- 負荷条件（同時接続数、RPS、データ量）
- 検証先（`sac-*` / `sts-*`）と運用監視（`opd-*`）

## 出力フォーマット（必須）

### YAML Frontmatter（必須）

```yaml
---
id: nfr-performance
type: architecture
title: 非機能要件: 性能
status: draft # draft / ready / deprecated
part_of: [nfr-index]
based_on: []
supersedes: []
---
```

### 本文の見出し（必須・順序固定）

1. 概要（対象・目的）
2. 適用範囲・負荷条件
3. 性能要件一覧
4. 測定・検証方法
5. 関連ドキュメント導線

## 記述ルール（重要）

- 要件一覧は表形式（`id` / `内容` / `指標` / `基準` / `備考`）で記載する。
- 指標は P95/P99、RPS、バッチ完了時間などを使い、単位を必ず付ける。
- 測定・検証方法は負荷試験と運用監視の双方で判定できる形にする。

## 禁止事項

- 負荷条件なしで閾値だけ書く。
- 平均値のみで評価する。
- 単位なしの数値を記載する。

## 最終チェック（自己検査）

- `id: nfr-performance` / `type` / `status` が設定されている
- 見出しが指定順序で揃っている
- 適用範囲に平常時/ピーク時条件がある
- 要件一覧に `指標` / `基準`（単位付き）がある
- 測定・検証方法に判定タイミングと判定者がある
