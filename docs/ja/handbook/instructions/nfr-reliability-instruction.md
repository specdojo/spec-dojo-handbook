# 非機能要件 / 信頼性 作成の指示テンプレート

以下のルールに従って、**非機能要件: 信頼性（`nfr-reliability`）** を 1 ファイル作成してください。出力は **Markdown** とします。

- 信頼性は障害抑止・誤動作防止・継続稼働を基準化します。
- 要件は対象・条件・指標・基準を明示し、合否判定可能にしてください。

## 事前に与えられる情報（入力）

- 対象システムと対象時間帯
- 障害履歴、業務重要度、除外条件（計画停止など）
- 検証先（`sac-*` / `sts-*`）と運用接続先（`opd-*` / `opr-*`）

## 出力フォーマット（必須）

### YAML Frontmatter（必須）

```yaml
---
id: nfr-reliability
type: architecture
title: 非機能要件: 信頼性
status: draft # draft / ready / deprecated
part_of: [nfr-index]
based_on: []
supersedes: []
---
```

### 本文の見出し（必須・順序固定）

1. 概要（対象・目的）
2. 適用範囲・前提条件
3. 信頼性要件一覧
4. 測定・検証方法
5. 関連ドキュメント導線

## 記述ルール（重要）

- 要件一覧は表形式（`id` / `内容` / `指標` / `基準` / `備考`）で記載する。
- 指標は障害率、MTBF、再試行成功率などを使い、単位付きで明記する。
- 測定・検証方法は「どこで」「いつ」「誰が判定するか」まで記載する。

## 禁止事項

- 「安定稼働する」だけで基準を書かない。
- 実装ライブラリ設定値を直接列挙しない。
- 検証方法を省略しない。

## 最終チェック（自己検査）

- `id: nfr-reliability` / `type` / `status` が設定されている
- 見出しが指定順序で揃っている
- 要件一覧に `id` / `指標` / `基準` がある
- 測定・検証方法に `判定タイミング` と `判定者` がある
- 関連ドキュメント導線が空欄でない
