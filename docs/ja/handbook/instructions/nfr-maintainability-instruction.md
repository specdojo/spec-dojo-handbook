# 非機能要件 / 保守性 作成の指示テンプレート

以下のルールに従って、**非機能要件: 保守性（`nfr-maintainability`）** を 1 ファイル作成してください。出力は **Markdown** とします。

- 保守性は変更容易性・障害解析容易性を定量化します。
- 実装方法ではなく、達成すべき運用品質を記述してください。

## 事前に与えられる情報（入力）

- 対象システム、運用時間、チーム体制
- 変更リードタイム、MTTR、切り分け目標
- 運用設計・運用手順（`opd-*` / `opr-*`）、受入条件（`sac-*`）

## 出力フォーマット（必須）

### YAML Frontmatter（必須）

```yaml
---
id: nfr-maintainability
type: architecture
title: 非機能要件: 保守性
status: draft # draft / ready / deprecated
part_of: [nfr-index]
based_on: []
supersedes: []
---
```

### 本文の見出し（必須・順序固定）

1. 概要（対象・目的）
2. 適用範囲・前提条件
3. 保守性要件一覧
4. 測定・検証方法
5. 関連ドキュメント導線

## 記述ルール（重要）

- 要件一覧は表形式（`id` / `内容` / `指標` / `基準` / `備考`）で記載する。
- 指標は MTTR、変更リードタイム、一次切り分け時間などを使用する。
- 測定・検証方法は監視記録・障害レポート・改善サイクルで追える形にする。

## 禁止事項

- 「保守しやすい」だけで閾値を示さない。
- 体制依存の一時運用を恒久要件化しない。
- 測定データの取得方法を書かない。

## 最終チェック（自己検査）

- `id: nfr-maintainability` / `type` / `status` が設定されている
- 見出しが指定順序で揃っている
- 要件一覧に `指標` / `基準` がある
- 測定・検証方法に判定タイミングと判定者がある
- 関連ドキュメント導線が空欄でない
