Non-Functional Requirements Usability Documentation Rules

本ドキュメントは、非機能要件のうち操作性（使いやすさ・誤操作防止・学習容易性）を定義する `nfr-usability` の作成ルールを定義する。
操作性要件は、利用者が誤りなく効率的に操作できることを検証可能に記述する。

## 1. 全体方針

- 操作性は完了率、誤操作率、習熟時間などで定義する。
- UI表現の詳細設計ではなく、利用結果としての品質基準を書く。
- 受入試験やユーザ検証で評価可能な要件にする。

## 2. 位置づけと用語定義（必要に応じて）

### 2.1. 位置づけ

- `nfr-usability`：操作性要求の正本
- `nfr-index`：NFR全体の入口
- `uis-*`：画面仕様の参照先
- `sac-*` / `sts-*`：操作性検証の受入・試験仕様

### 2.2. 用語定義

| 用語         | 定義                                     |
| ------------ | ---------------------------------------- |
| 操作性       | 利用者が目的タスクを迷わず完了できる性質 |
| タスク完了率 | 指定タスクを完了できた割合               |
| 誤操作率     | 望まない操作を実行した割合               |

## 3. ファイル命名・ID規則

- ルールドキュメントIDは `nfr-usability-rules`。
- 生成対象ドキュメントIDは `nfr-usability` を推奨する。
- 要件IDは `nfr-usb-<連番>` を推奨する。

## 4. 推奨 Frontmatter 項目

### 4.1. 設定内容

Frontmatter は共通スキーマに従います（参照: [shared/schemas/spec-frontmatter.schema.yaml](../../../../shared/schemas/spec-frontmatter.schema.yaml) / [meta-document-metadata-rules.md](meta-document-metadata-rules.md)）。

| 項目    | 説明                                | 必須 |
| ------- | ----------------------------------- | ---- |
| id      | `nfr-usability`                     | ○    |
| type    | `architecture` など共通スキーマ準拠 | ○    |
| title   | 非機能要件: 操作性                  | ○    |
| status  | `draft` / `ready` / `deprecated`    | ○    |
| part_of | `nfr-index`                         | 任意 |

### 4.2. 推奨ルール

- ユーザ属性（熟練度、利用環境）を前提として明示する。

## 5. 本文構成（標準テンプレ）

`nfr-usability` は以下の見出し構成を **順序固定** で配置する。

| 番号 | 見出し               | 必須 |
| ---- | -------------------- | ---- |
| 1    | 概要（対象・目的）   | ○    |
| 2    | 適用範囲・利用者前提 | ○    |
| 3    | 操作性要件一覧       | ○    |
| 4    | 測定・検証方法       | ○    |
| 5    | 関連ドキュメント導線 | ○    |

## 6. 記述ガイド

### 6.1. 概要（対象・目的）

生成する本文の見出しは **## 1. 概要（対象・目的）**

- 対象利用者と達成すべき操作品質を記述する。

### 6.2. 適用範囲・利用者前提

生成する本文の見出しは **## 2. 適用範囲・利用者前提**

- 対象画面、対象業務、利用者の熟練度前提を示す。

### 6.3. 操作性要件一覧

生成する本文の見出しは **## 3. 操作性要件一覧**

- 表形式（`id` / `内容` / `指標` / `基準` / `備考`）で記述する。
- 指標例：タスク完了率、平均完了時間、誤操作率。

### 6.4. 測定・検証方法

生成する本文の見出しは **## 4. 測定・検証方法**

- ユーザ検証、受入試験、ログ分析で判定する。

### 6.5. 関連ドキュメント導線

生成する本文の見出しは **## 5. 関連ドキュメント導線**

- `uis-*`、`sac-*`、`sts-*`、`adr-*` への導線を記載する。

## 7. 禁止事項

| 禁止事項                             | 理由                   |
| ------------------------------------ | ---------------------- |
| 「使いやすい」だけで指標を定義しない | 判定不能のため         |
| 画面詳細仕様を丸ごと再掲する         | 役割重複になるため     |
| 利用者前提を書かない                 | 評価条件が揃わないため |

## 8. サンプル

注：以下はルール文書内の例示です。生成する `nfr-usability` では `## 1...` から始まります。

```yaml
---
id: nfr-usability
type: architecture
title: 非機能要件: 操作性
status: draft
part_of: [nfr-index]
based_on: []
supersedes: []
---
```

### 8.1. 概要（対象・目的）（例）

本書は、GCS 基幹システムにおいて操作性（使いやすさ・誤操作防止・学習容易性）に
関する非機能要件を定義する。
利用者が業務タスクを迷わず完了でき、誤操作を最小化することを
検証可能な基準で管理し、業務効率の維持・改善を目的とする。

### 8.2. 適用範囲・利用者前提（例）

| 項目         | 内容                                                 |
| ------------ | ---------------------------------------------------- |
| 対象画面     | GCS Web UI（受注管理・在庫照会・会計ダッシュボード） |
| 対象業務     | 受注登録・在庫確認・請求書発行・売上確定             |
| 利用者属性   | 業務担当者（PC・ブラウザ操作に慣れた一般事務職）     |
| 新任利用者   | 業務知識あり・システム未経験                         |
| 対象デバイス | デスクトップ PC（Chrome 最新版 / 解像度 1920×1080）  |

### 8.3. 操作性要件一覧（例）

| id          | 内容                                       | 指標                 | 基準       | 備考                     |
| ----------- | ------------------------------------------ | -------------------- | ---------- | ------------------------ |
| nfr-usb-001 | 主要業務タスクを迷わず完了できること       | タスク完了率         | 95% 以上   | 新任利用者対象           |
| nfr-usb-002 | 誤操作を基準以下に抑える                   | 誤操作率             | 2% 以下    | 取消導線を提供           |
| nfr-usb-003 | 新任利用者が基本操作を習得できること       | 習得時間             | 2 時間以内 | オンボーディングガイド付 |
| nfr-usb-004 | 受注登録は基準時間内に完了できること       | 平均完了時間         | 3 分以内   | 入力フィールド 15 件以内 |
| nfr-usb-005 | エラーメッセージは対処方法を含むこと       | 対処方法含有率       | 100%       | 「…してください」形式    |
| nfr-usb-006 | 破壊的操作には確認ダイアログを表示すること | 確認ダイアログ網羅率 | 100%       | 削除・取消・上書き       |
| nfr-usb-007 | キーボードのみで主要操作が完了できること   | キーボード操作完結率 | 90% 以上   | アクセシビリティ対応     |

### 8.4. 測定・検証方法（例）

| 要件ID      | 測定方法                                     | 判定タイミング        | 判定者        |
| ----------- | -------------------------------------------- | --------------------- | ------------- |
| nfr-usb-001 | ユーザビリティテスト（5 名×主要タスク）      | リリース前受入試験    | UX / QA       |
| nfr-usb-002 | 操作ログから誤操作イベントを集計             | 月次集計              | UX            |
| nfr-usb-003 | 新任利用者オンボーディング後の操作確認テスト | リリース後 1 か月以内 | UX / 業務部門 |
| nfr-usb-004 | タスク計測（受注登録開始〜完了までの時間）   | ユーザビリティテスト  | UX / QA       |
| nfr-usb-005 | エラーメッセージ一覧の対処方法含有率を確認   | リリース前レビュー    | QA            |
| nfr-usb-006 | 破壊的操作一覧と確認ダイアログ実装状況の突合 | リリース前レビュー    | QA            |
| nfr-usb-007 | キーボード操作で主要タスクを実行し完結を確認 | リリース前試験        | QA            |

### 8.5. 関連ドキュメント導線（例）

| 種別         | ドキュメントID | 目的                         | 備考 |
| ------------ | -------------- | ---------------------------- | ---- |
| NFR 入口     | nfr-index      | 非機能要件全体の俯瞰         | 必須 |
| 受入条件     | sac-\*         | 操作性要件の合否判定基準     | 必須 |
| システム試験 | sts-\*         | 操作性観点の E2E 試験        | 必須 |
| 画面仕様     | uis-\*         | 画面レイアウト・操作フロー   | 必須 |
| 判断記録     | adr-\*         | 操作性に関する設計判断の追跡 | 任意 |
