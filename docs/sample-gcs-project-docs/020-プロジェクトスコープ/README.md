# プロジェクトスコープ

プロジェクトの範囲（対象／対象外）、主要な前提・制約、納品物などを明確にして、関係者間の合意を得ることを目的とします。

## ドキュメントの目的

- 本ドキュメントは、プロジェクトの「何を行うか」「何を行わないか」を明確化し、関係者（事業、開発、QA、運用）間の合意形成を支援します。
- 要件定義や計画策定、見積もり、テスト範囲決定、移行計画の基礎情報を提供します。

## ドキュメントの内容

本READMEには、少なくとも以下を含めます。

- 対象：システムや業務で明確に取り扱う範囲
- 対象外：今回のプロジェクトで扱わない範囲
- 主要機能・成果物：実装対象の主要な機能や成果物の一覧
- 前提条件・制約：外部依存、技術的制約、想定ユーザ数など
- 関係者：事業側の主要なステークホルダ

## なぜこのドキュメントが必要か

- 範囲を早期に合意することで、後工程での手戻りや仕様変更によるコストを削減します。
- 見積もり、スケジュール、品質基準（テスト範囲）の基礎情報になるため、計画の精度を高めます。
- スコープ外の要求を切り分けることで、契約・予算管理が容易になります。

## このドキュメントがないとどう困るか

- 期待値の不一致が発生しやすく、機能追加や仕様変更が頻発して工数増加につながる。
- テストや受け入れの範囲が不明瞭になり、リリース後の不具合や受け入れ拒否の原因となる。
- 外部依存（他システム、外部ベンダー）との調整が遅れ、プロジェクト全体の遅延を招く。

## サンプル

以下は本ドキュメントの簡易サンプルです。実プロジェクトではこの骨子を埋めてください。

- 対象（In Scope）:
  - ユーザ管理（登録/編集/削除）
  - 注文管理（注文作成、注文履歴表示、ステータス更新）
  - レポート（売上集計、商品別ランキング）

- 対象外（Out of Scope）:
  - 決済処理の実装（外部決済サービスを利用、接続仕様は別途調整）
  - 倉庫在庫管理システムとの連携（別プロジェクト）

---

注: 上記は例です。プロジェクト固有の条件（法規制、セキュリティ要件、パフォーマンス要件等）を必ず反映してください。
